# RAG

> 技术背景

这就要从传统 LLM 的局限性说起：

1. 知识局限性：LLM 知识认知来源于训练数据，而现在大多数训练数据都是网络公开的数据，不能涵盖实时或者私有专有领域的数据，所以一旦询问训练集外的全新领域模型就会无从下手。

2. 幻觉性：深度学习模型的底层原理大多基于数学概率，而模型的输出实质上是一系列数值运算的结果，所以在某些场景会出现一本正经回答错误内容特别是在一些不擅长的任务场景，缺少事实论证。

RAG（Retrieval Augmented Generation. 检索增强生成）是<u>*一种提高文本生成质量和相关性的技术，通过引入外部知识库让 LLM 在生成回答前先检索相关信息</u>*。有了 RAG 以后LLM 生成文本前，先通过检索机制从外部知识库中动态获取相关信息，并将这些“参考资料”融入生成过程从而显著提高回答的**时效性**（知识库不断迭代更新）、**准确性**（有事实的依据）。



# RAG 工作原理

![image-20251202001106209](./assets/rag%20workflow.png)

流程上分为几个步骤：

1. 数据提取：通常是一些 PDF、DOCX 文档作为数据源，提取里边的文本作为数据输入。
2. 文本分块：将提取到的原始文本数据分块。
3. 向量嵌入&存储：处理完后的向量矩阵需要存储到向量数据库中（元数据）
4. 向量检索：在拿到 Query 后需要在数据库中找到相似度最高的矩阵块，此时先要把 Query 也转化为向量矩阵，然后再采用像余弦相似度等距离算法去计算矩阵之间的距离召回前 n 个矩阵块。
5. 文本生成：将用户问题、检索文档片段、角色设定等相关信息整合，引导进行文本生成。



## 数据分块

> Q：为什么需要分块？基于原始文本直接嵌入不是更省事

1. 向量化过程需要嵌入模型（Embedding Model）介入，而嵌入模型本身有严格的输入长度限制（上下文窗口），如果不加分割直接传入源文本会导致超出窗口的信息丢失，结果就是会破环文本语义
2. 检索完成后需要将结果返回给 LLM，而 LLM 本身同样有上下文窗口限制，如果输出文本过长的话会导致其他信息（prompt、用户问题等）丢失，影响生成质量



> Q：为什么分块不是越大越好？上下文窗口既然有最大限制那直接拉满是不是更好

1. 检索到的分块最后是需要给到 LLM，当 LLM 处理非常长的上下文时，倾向于记住开头和结尾的信息，而忽略中间部分，最终导致模型无法有效提取关键信息，反而降低生成质量
2. 太长的文本主题/语义稀疏，在进行检索的时候难以通过有效的手段区分



> 文本分块策略

下边是以 LangChain 提供的分块策略来说明几种常见的分块策略

- 固定分块（`CharacterTextSplitter` ）：核心的逻辑是根据传入的分隔符切割文本；但这个方法有个弊端是再识别到超长后没有进行有效处理，仅仅用 `console.warn` 提示后还是会将分块列表合并

- 递归字符分块（`RecursiveCharacterTextSplitter`）：核心的思路是保证最后每个分片尽可能不超出 `chunkSize`
  - **寻找有效分隔符**: 从分隔符列表中从前到后遍历，找到第一个在当前文本中**存在**的分隔符
  - **切分与分类处理**: 使用选定的分隔符切分文本，然后遍历所有片段：
  - **超长处理**：当分块长度超过最大限制时，会递归使用下一个存在的分隔符进行切割

##  向量嵌入

> 1. 什么是向量嵌入？为什么需要在 RAG 里面引入这个技术？
> 2. 向量嵌入的输入 / 输出
> 3. embedding 模型的发展历史

为什么需要在 RAG 中引入这项技术呢？RAG 检索技术的核心做文本相似度比较，从人的角度来看我们可以通过自身的理解来判断不同输入语义上是否相似，但是从机器的角度来看，它们只认识数字不认识文本，无法处理相似性判断的任务，所以就需要我们把输入转化为机器可以理解的形式。而向量嵌入的核心目标是**弥合自然语言（符号世界）与数学模型（向量空间）之间的鸿沟**，将真实世界中复杂、高维的数据对象（如文本、图像、音频、视频等）转换为数学上易于处理的、相对低维、稠密的连续数值向量

![Embedding 过程示意图](./assets/3_1_1.webp)

在经过转换后其实向量就是在一个空间坐标系里边的坐标点，是否相似就变为了坐标点之间的距离是否相近，抽象的相似性判断就被具体化为数值上的计算（余弦相似度 、点积、欧式距离），而在这过程中扮演着重要角色的就是把文本转化为向量的 **<u>*Embedding 模型*</u>**。Embedding 模型的演进路径大致可以分为几个关键阶段：

> 词向量生成发展历程
>
> 1. 独热编码 One-Hot Encoding
> 2. 词袋模型 Bag-of-Words
> 3. N-gram 模型
> 4. TF-IDF
> 5. Word2Vec
> 6. Transformer
> 7. BERT

- 静态词嵌入：比较典型的代表是 Word2Vec（2013）、GloVe（2014），核心的逻辑是通过学习为每个单词生成一个固定与上下文无关的向量（神经网络），但缺点也比较明显，针对相同的单词会生成相同的向量，无法处理词语多义性
- 动态上下文嵌入：2017年，`Transformer` 架构的诞生带来了自注意力机制（Self-Attention），它允许模型在生成一个词的向量时，动态地考虑句子中所有其他词的影响。基于此，2018年 `BERT` 模型利用 `Transformer` 编码器，通过掩码语言模型（MLM）等自监督任务进行预训练，生成了深度上下文相关的嵌入，解决了静态词嵌入过程中词语多义问题。
- 多模态嵌入：现实世界的信息不仅仅只有文本，还包括图像、音频、视频等多模态的数据，因此逐渐涌现了支持多模态嵌入的模型，包括 OpenAI 的 CLIP、BAAI的 BGE-M3（20240205）

>💡 如何选择合适的 Embedding 模型

可查看公开的模型排行榜 [Hugging Face Embedding Leaderboard](https://huggingface.co/spaces/mteb/leaderboard)，根据实际的任务类型、处理的文本语种、运行环境等等选择合适的模型

#  RAG 局限性

而在实际业务的应用中传统 RAG 技术还是面临一些挑战：

1. 检索质量
   1. 文本语义不唯一：比如苹果在现实中可以理解为 “苹果笔记本” 或者是 “水果中的苹果”，而在向量化的过程中其实无法区分相同词组的不同含义
   2. 输入 query 多元：当用户的输入由短语、关键词变为长句甚至是比较口语化的输入时，单纯通过文本块相似计算似乎变得不再精确

2. 增强过程
   1. 在召回阶段如果存在多个相似的文本块缺乏有效衡量区分手段（无法做取舍）
   2. 单次任务检索失败缺少有效的 “换位思考”
3. 生成过程
   1. LLM 过度依赖增强过程返回的信息，一旦增强内容不准确或者有误无法有效鉴别



# RAG 演进史

为了解决传统 RAG 应用过程中的问题，业界也在不断做技术演进，提出不同的技术来优化流程

![img](https://datawhalechina.github.io/all-in-rag/chapter1/images/1_1_2.png)

不同的优化技术其实都是围绕三个方向：

- 如何保证更好的嵌入质量
- 如何保证更高的检索召回率
- 如何保证更好的生成质量



## Advanced RAG

> 引入预检索（如查询重写/扩展）、后检索（结果重排序、摘要压缩）

https://blog.csdn.net/hugo_lei/article/details/139215091



## Modular RAG

> 将整个流程拆分为可插拔的七大模块（如 Indexing、Pre Retrieval、Retrieval、Post Retrieval、Memory、Generation、Orchestration），支持检索、重排序、压缩、生成等功能模块化组合，并引入路由、调度、融合算子，实现灵活扩展和多种 RAG 模式（线性、条件、分支、循环）



## Agentic RAG

> 智能体架构，自主判断何时检索、何时生成，支持多轮迭代和任务自校正

> 技术本身都是确保做好以下三个事情
>
> 1. 怎么切割的好
> 2. 怎么检索的准
> 3. 怎么生成得好

- 嵌入阶段

  - 普通文本切割劣势较大，通常会导致切割的片段语义不完整

  - 按句子、语义切割，能较大程度保证切割的块语义是完整；但缺点就是召回率比较低

  - 上下文存储：在向量化的时候将前后上下文也一并存储

  - 主题标识：为每一个文本块生成一个主题；即使有相同的单元也能通过主题去去分

  - 为文本块生成相关问题，一起嵌入存储，可大幅提升召回率

- 检索阶段

  - 查询转换；在查询阶段对用户的查询词语进行转换（具像化）

  - Reranker：针对根据相似匹配返回的结果进行二次重排（主题、LLM 思考）

  - 关联段落提取：得到的 TopK 后判断是否存在在原文中连续的片段，如果有拼接返回

- 生成阶段
  - 上下文压缩：针对得到的筛选结果进行概括性总结（去除一些多余文本）
  - 反馈循环



- RAG 分类
  - 适应性 RAG：判断用户询问的类型去使用不同策略处理，可避免不必要检索
    - 事实型：文档查询
    - 分析型：文档查询 + 主观回答
    - 意见型：主管回答
  - Self RAG：自我反思各个步骤，容错率高
  - Fusion RAG（融合 RAG）：检索阶段使用关键词检索 + 语义检索，结合两种检索结果最后得到最优答案
  - Graph RAG（知识图谱）：为每个文本分块提取实体 + 关系，能极大提升检索效率
  - Hierarchy RAG（分层 RAG）：为每个文本分块生成摘要，检索阶段先检索摘要筛选后再检索文本
  - HyDE（假设性 RAG）：为问题先生成一个预设的答案，将预设答案嵌入检索（答案与原文可能更相似）
  - Crag（纠错型 RAG）：在得到结果后判断相关性决定是否要进行其他工具调用（网络搜索）
